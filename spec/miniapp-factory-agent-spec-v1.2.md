## 1. 背景と事例調査（このアーキテクチャの根拠）

### 1.1 何を作るか（1行）
**「ユーザー入力または公式の計算式だけで完結する、単機能で"実際に動く"Webツール」を大量に作り、ロングテール検索を取り、広告/アフィリで現金化する量産・運用基盤。**

### 1.2 参照事例（開発プロセスの設計根拠）

調査した実在の運用事例から、本アーキテクチャの設計判断を導いている。

- **Omni Calculator（最重要の成功事例）**
  - 3,700本超の電卓を擁し、月間2,300万訪問規模。収益モデルは Google AdSense（後に Playwire / Google Ad Manager を併用）。技術は React + Python(Django) + AWS。
  - 各電卓ページは **「問題提起 → 計算式の説明 → 計算の手順」** という共通パターン。被リンクが自然に集まり（Healthline 等の大手が電卓を引用）、累積で巨大な権威を獲得。
  - **決定的な学び**: Google の Helpful Content 系アップデートで沈むどころか伸びた。理由は **各ページが"薄い記事"ではなく"実際に動く機能"を提供しているから**。→ 工場のユニットは必ず「動く」こと。これが後述のスパム判定回避の核。

- **ShipFast / Marc Lou（"量産の型"の成功事例）**
  - 27本の試作で繰り返した定型処理（認証・課金・メール・UI部品・LP・価格・オンボーディング）を **1つのボイラープレートに束ね**、以後の出荷速度を劇的に上げた。
  - **無料ツールを"入口"にしてソフトセルで有料へ**誘導する導線（無料の単機能ツール → 有料プロダクト）。これは v3 §3 の **B→A 統合パターンそのもの**。
  - **PMF がなければ素早く撤退**、ポートフォリオで賭ける。
  - **学び**: 量産の本質は「パイプラインを一度作って使い回す」こと（v3 §6）。工場は再利用可能な部品（UI/SEO/課金/アフィリ導線）を中核に持つ。

- **Pieter Levels（高自動化ポートフォリオの事例）**
  - 複数プロダクトを高自動化＋Build in Public で運用。**小さく多数の賭けを並走**。
  - **学び**: 人間の関与を最小化する自動運用ループ（本仕様書 §7）の妥当性。

### 1.3 ★最重要リスク：Google「スケール化されたコンテンツの不正利用 / 誘導ページ」

量産は **両刃の剣**。2024〜2026 の一連のスパムアップデートで、Google は以下を明確に処罰している：

- **Scaled content abuse（大量生成された低価値ページ）** … 「**手段（AI/自動化）ではなく意図と価値**」で判定。人間が2,000本の量産テンプレを書いても同じく違反。
- **Doorway pages（誘導ページ）** … キーワードや地名だけ差し替えた似たページを大量生成 = 処罰。
- 検出は SpamBrain（AI）で、**サイト/ドメイン横断のパターン**まで見る。手動対策＋アルゴリズム降格の両方。

**→ 工場の生死を分ける防御線（本仕様書全体で機械的に強制する）:**
1. **ユニットは"記事"ではなく"動く機能"**（決定論的な計算/換算/見積/逆算）。テキストの薄い差し替え量産は **禁止**。
2. **各ページに実質的な独自価値**（固有の計算ロジック・固有のUX・公式の出典・固有のFAQ）。Omni / Zillow 型（実データ・実機能）は生き残り、薄いテンプレは沈む、という線を死守。
3. **量より質のスケール**。出すこと自体は速くてよいが、品質下限（§5.2）を割ったものは公開しない。

> この §1.3 は工場の「やってはいけないこと」の根幹。後述の **quality-gate サブエージェント**（§3.2）と **quality-rubric スキル**（§3.3）でリリース前に毎回自動チェックする。

## 2. 設計思想（工場の根本原則）

| # | 原則 | 含意 |
|---|---|---|
| P1 | **ユニットは"動く"単機能ツール** | コンテンツではなく機能で勝つ。AI ランタイム不要・スクレイピング不要・低保守（v3 §2 ハードゲート）。アプリ実行時に AI は使わない（決定論コア）。AI を使うのは**ビルド時だけ**。**原則は静的・DBなし**だが、特定アプリで DB/動的が真に必要なら例外的に足せる（既定ではない・必須でもない、§4.A.1++）。 |
| P2 | **工場 = 1モノレポ + 再利用部品** | UI部品・SEO雛形・課金/アフィリ導線・品質ゲートを一度作り使い回す（v3 §6、ShipFast の教訓）。 |
| P3 | **出荷 = 検証、ただし品質下限は不可侵** | 事前リサーチで絞り込みすぎない（v3 §0,§5）。が、§1.3 のスパム線を割るなら出さない。「速く・かつ薄くない」。 |
| P4 | **人間は意思決定とレビューに集中** | 朝＝音声で指示出し（issue化）／日中＝エージェントが回す／夜＝レビュー&マージ。可処分時間 30分/日・月15h が最大制約（memo §1）。 |
| P5 | **2つのレールを段階的に重ねる** | レールB（無料ツール→SEO→広告/アフィリ）を主力で量産。トラフィックが溜まったら同業界にレールA（少額課金の一段深いツール）を被せる（v3 §3、Marc Lou 導線）。 |
| P6 | **当事者性はレールで使い分け** | レールBは公式の計算式で作れるので当事者性不要 → 任意の業界へ横展開可。レールA・初期の"型作り"は当事者領域（投資/コンサル/AIエージェント）で速く回す（v3 §1, §7）。 |
| P7 | **ガードレールはコードで強制** | 規制・スパム・広告ポリシーは「気をつける」ではなく **リリースブロッカー** として実装（§9）。 |
| P8 | **トークン予算内で完結** | 探索は Haiku、実装は Sonnet、重い出力はサブエージェントに隔離（§3, §7）。サブスク枠内に収める（memo §3.3）。 |

## 3. システムアーキテクチャ（Claude Code 上の構成）

工場は **1つのオーケストレーター + 役割別サブエージェント + スキル群 + フック + 状態ファイル** で構成する。Claude Code のネイティブ機能（subagents / skills / hooks / plugins）に素直にマッピングする。

```
┌─────────────────────────────────────────────────────────────┐
│  factory-orchestrator  (メインセッション or --agent で起動)   │
│  ・バックログ(issue)を読み、各ステージを適切なサブエージェントへ委譲 │
│  ・夜のレビュー時は人間に要約を返す                              │
└───────────┬──────────────┬───────────────┬──────────────────┘
            │              │               │
   ┌────────▼───┐  ┌───────▼──────┐  ┌─────▼────────┐  ... 
   │opportunity │  │ tool-builder │  │seo-optimizer │   各サブエージェントは
   │  -scout    │  │              │  │              │   独立コンテキスト・
   └────────────┘  └──────────────┘  └──────────────┘   tools制限・model指定
   ┌────────────┐  ┌──────────────┐  ┌──────────────┐
   │quality-gate│  │  publisher   │  │analytics-    │
   │(blocker)   │  │(人間承認必須) │  │  analyst     │
   └────────────┘  └──────────────┘  └──────────────┘

   横断スキル(.claude/skills/): tool-spec / tool-scaffold / seo-checklist /
   monetization / quality-rubric / jp-compliance / publish-runbook

   フック(settings.json & frontmatter): デプロイ前の品質ゲート強制 /
   破壊的コマンドのブロック / ビルド後の Lighthouse・リンク切れ検査 /
   SubagentStop でレジストリ更新

   状態(リポジトリにコミット): registry.json / metrics/ / decisions.log /
   backlog/ (issue)
```

### 3.1 オーケストレーター
- **モデル: Opus**。工場全体を管理し、状態（registry/metrics/backlog）を読み、どのサブエージェントへ委譲するかを判断し、夜のレビューを統括する"頭脳"。ここはケチらず最上位モデルを使う（§3.2 末尾「モデル配分の方針」参照）。
- 起動: `claude --agent factory-orchestrator`（セッション全体をこの人格で動かす）。`.claude/settings.json` の `agent` で既定化してよい。
- 役割: `backlog/` の issue を読み、各 issue のステージ（§6）に応じて適切なサブエージェントへ委譲。`Agent(...)` の allowlist で **呼べるサブエージェントを限定**（暴走防止）。
- 出力: 夜のレビュー時、人間に「今日やったこと・要承認の項目（公開/課金/権限）・次の判断ポイント」を **1メッセージで簡潔に**返す。

### 3.2 サブエージェント群

各サブエージェントは `.claude/agents/<name>.md`（プロジェクトスコープ、Git管理）で定義。frontmatter で `tools` / `model` / `memory` を絞る。**必須フィールドは `name` と `description`。** `description` に「いつ委譲するか」を明確に書く（自動委譲の精度はここで決まる）。

| サブエージェント | 役割 | 推奨 model | tools（例） | memory | 備考 |
|---|---|---|---|---|---|
| `opportunity-scout` | 業界×「面倒な計算/換算/見積/手数料/按分/逆算」の棚卸し、既存ツールの有無・キーワード需要・アフィリ在庫の確認、機会を issue 化 | `haiku`（探索は安く） | Read, Grep, Glob, WebSearch, WebFetch, Write | `project` | 出力は §付録の issue テンプレに整形。公式計算式の出典URLも収集。 |
| `tool-builder` | 1ユニットの実装（入力UI・**決定論的計算ロジック**・単体テスト・メタ）。`tool-spec`/`tool-scaffold` スキルを使う | `sonnet` | Read, Write, Edit, Bash, Glob, Grep | `project` | アプリ内に AI/外部API/スクレイピングを **入れない**。logic は純関数＋テスト必須。 |
| `seo-optimizer` | title/description/見出し構造/FAQ/構造化データ(schema)/内部リンク/サイトマップ更新 | `sonnet` | Read, Write, Edit, Glob, Grep | `project` | `seo-checklist` スキル準拠。誘導ページにしない（§1.3）。 |
| `quality-gate` | **リリースブロッカー＆バグ/品質確認**。§1.3 スパム自己診断・E-E-A-T・必須ページ・規制(§9)・重複/類似度・計算ロジックの妥当性・Core Web Vitals を検査し pass/fail を返す | **`opus`** | Read, Grep, Glob, Bash | `project` | fail なら publisher へ進ませない。**バグ・規制抵触・薄さの最終砦なので最上位モデル**。`quality-rubric`+`jp-compliance` スキル使用。 |
| `publisher` | ビルド → デプロイ手順の実行。ただし **本番デプロイ・独自ドメイン公開・課金有効化は人間承認**（§7.3） | `sonnet` | Read, Bash | （なし） | `publish-runbook` スキル準拠。破壊的操作はフックでブロック。 |
| `analytics-analyst` | GSC/GA4/AdSense・ASP のデータを集計し、勝ち筋/負け筋を判定 → 改善 or 撤退の issue を生成 | `sonnet` | Read, Write, Bash, WebFetch（or MCP） | `project` | 「出荷=検証」を計測で閉じる（§8）。 |

> **モデル配分の方針（重要）**: 「管理・判断・最終品質」は最上位モデル、「大量に回す作業」は安いモデル、という配分にする。
> - **Opus**: `factory-orchestrator`（全体管理・委譲判断・夜レビュー統括）と `quality-gate`（バグ/規制/薄さの最終砦）。**ここはケチらない。** 小さなツールでも、それを"束ねて管理する層"と"出荷可否を決める層"の判断ミスは工場全体に波及するため。
> - **Sonnet**: `tool-builder`・`seo-optimizer`・`publisher`（1本ずつの実装・SEO・公開。小機能の生成はSonnetで十分）。
> - **Haiku**: `opportunity-scout`（探索・調査は最も安いモデルで量をこなす）。
> - **任意**: 月次の戦略レビュー（勝ち筋への資源集中・撤退判断）を深くやらせたい回だけ、`analytics-analyst` を一時的に Opus に上げてもよい（`CLAUDE_CODE_SUBAGENT_MODEL` か呼び出し時の `model` 指定で上書き可能）。
> モデルは frontmatter の `model` にエイリアス（`opus`/`sonnet`/`haiku`）で書く。バージョン更新に強い。

> **メモリの使い方**: 各サブエージェントの `memory: project` で、横展開時に効く知見（効いた title 型・効いた内部リンク構造・各業界の公式計算式の所在・落ちやすい品質項目）を蓄積する。Git 管理され、横展開の資産になる。

### 3.3 スキル群（`.claude/skills/<name>/SKILL.md`）

スキルは「Claude にタスクの**やり方**を教える」再利用部品。**Progressive disclosure** を守る：SKILL.md 本体は ~500行以内に収め、重い内容（チェックリスト全文・テンプレ・公式集）は `references/` に置き、必要時のみ読み込ませる。決定論的な処理は `scripts/`（コンテキストを消費しない実行可能スクリプト）に逃がす。

| スキル | 何を教えるか | references/ scripts/ の例 |
|---|---|---|
| `tool-spec` | 1ユニットの仕様書テンプレ（入力/出力/計算式/出典/想定キーワード/レール/アフィリ先/想定FAQ） | `references/spec-template.md` |
| `tool-scaffold` | 雛形生成手順：ページ・`logic.ts`(純関数)・`logic.test.ts`・`meta.json`・`schema.json` を所定の構成で作る | `scripts/scaffold.ts`（雛形生成）、`references/unit-anatomy.md` |
| `seo-checklist` | オンページSEO：title/meta/H1-H3/イントロ(問題提起)/手順/FAQ/schema(WebApplication+FAQPage+HowTo)/内部リンク/canonical/sitemap | `references/onpage-checklist.md`, `references/schema-snippets.md` |
| `monetization` | 広告枠の配置（不快でない位置・CLS回避）、アフィリリンクの設置、**PR表記の必須化**（ステマ規制）、レール別の現金化先 | `references/ad-slots.md`, `references/jp-asp-list.md`, `references/disclosure.md` |
| `quality-rubric` | §1.3 スパム自己診断・E-E-A-T・必須ページ(About/Contact/Privacy/Terms)・低価値判定・類似度しきい値 | `references/spam-self-audit.md`, `references/eeat.md` |
| `jp-compliance` | 規制ガードレール（投資助言/税務/行政/医療等の独占業務・助言に踏み込まない、計算・記録・通知に留める） | `references/regulatory-guardrails.md` |
| `publish-runbook` | ビルド→プレビュー→人間承認→本番デプロイ→indexリクエスト→registry更新の手順 | `references/deploy-steps.md` |

> スキルとサブエージェントの併用：`tool-builder` の frontmatter `skills:` に `tool-spec`, `tool-scaffold` をプリロードしておくと、起動時に手順が文脈に入る（毎回 Skill 探索する手間が省ける）。

### 3.4 フック（`settings.json` とサブエージェント frontmatter）

ガードレールと品質を **人間/AIの善意に依存せず強制**するための仕組み。

- **`PreToolUse`（Bash）= 破壊的/越権操作のブロック**：本番デプロイ系コマンド・`rm -rf`・権限変更・秘密情報の流出につながる操作を検出したら exit code 2 でブロックし、stderr で理由を返す。
- **`PreToolUse`（publisher のデプロイ前）= 品質ゲート強制**：対象ユニットの quality-gate が pass 済みであることを確認（`registry.json` のフラグを参照）。未passならブロック。
- **`PostToolUse`（Edit/Write 後）= 自動検査**：`logic.test.ts` 実行、Lighthouse/Core Web Vitals 計測、リンク切れ・schema 妥当性チェックを走らせ、失敗を可視化。
- **`SubagentStop`**：`tool-builder`/`publisher` 完了時に `registry.json` と `metrics/` を更新するスクリプトを実行。

> exit code 2 = ブロック＆理由をモデルに返す、という Claude Code フックの仕様を使う。検証スクリプトは `scripts/` に置く。

### 3.5 状態（リポジトリにコミットして永続化）
- `registry.json` … 全ユニットの台帳（§8.1 スキーマ）。
- `metrics/YYYY-MM.json` … GSC/GA/収益のスナップショット。
- `decisions.log` … 撤退/強化/横展開などの意思決定ログ（追記式）。
- `backlog/*.md` … issue（§付録テンプレ）。ステージ・状態を front-matter で持つ。

## 4. リポジトリ構成（具体）

## 4.A ホスティング・公開・管理の技術仕様（どこに・どう公開するか）

> エージェントが最終的に作るのは「**1つのドメイン上に、多数のミニアプリを各ページとして並べた集約サイト**」。その公開先・公開手順・管理方法を明示する。

### 4.A.1 公開先（既定：Cloudflare Pages 無料枠）
- **既定 = Cloudflare Pages（無料）**。理由：**帯域無制限・クレカ不要・商用利用OK（広告/アフィリOK）・無料SSL・独自ドメイン対応・Gitプッシュで自動ビルド/デプロイ・PRごとのプレビュー・無料の簡易アクセス解析**。SEOツール集約サイトに必要な条件をゼロ円で満たす。
- **使わないもの = Vercel 無料(Hobby)**。Hobby は **非商用・個人利用限定**で、**広告（AdSense含む）や訪問者からの金銭授受は規約違反**（公式 Fair Use ガイドライン）。さらに `*.vercel.app` サブドメインは AdSense の所有権確認が通りにくい。→ 広告で回す工場では不可。
- **静的書き出しを既定にする理由（費用の安全性）**: レールBのツールは計算がブラウザ側で完結する純粋な静的サイトにできる。SEOで露出が増えると**AIクローラ等のbotアクセスが激増**しがちで、サーバー従量課金のホスト（Vercelの新しい従量モデル等）では**bot流入だけで費用が爆発した実例**がある。静的配信＋Cloudflareの無制限帯域なら、この費用リスクをほぼゼロにできる。
- **代替（任意）**: DXを優先して Vercel を使いたいなら **Vercel Pro（$20/月）＋独自ドメイン**が必要（無料では広告不可のため）。Netlify 等でも可だが、まずは Cloudflare Pages 無料が費用最小。

### 4.A.1+ ドメインは買わないとダメ？（無料で始められる／AdSense だけは別）
切り分けが大事：
- **ホスティング自体は完全無料**。`yourname.pages.dev`（Cloudflare）や `yourname.github.io`（GitHub Pages）などの**無料サブドメイン**で公開できる。**ドメイン購入は必須ではない。**
- **ただし Google AdSense は"自分が所有するドメイン"が必要**。`*.pages.dev` / `*.github.io` / `*.vercel.app` / `*.wordpress.com` などの無料サブドメインは、**親ドメインを各社が所有**しており、あなたはそこに AdSense コードを置けないため、**無料サブドメインのままでは AdSense の審査・設定を完了できない**。
- つまり **年¥1,000〜2,000のドメイン代は「ホスティング費用」ではなく実質「AdSense の入場料」**。サーバー・帯域・SSL・ビルドは全部無料。

→ 選べる2つの道（どちらでも可）：
- **(推奨) ドメインを1つだけ買う**：年¥1,000〜2,000。これで AdSense・信頼性・ブランドが一括解決。唯一かつ最小の固定費。
- **(完全無料で始める) 無料サブドメイン＋アフィリ先行**：まず `*.pages.dev` 等にアフィリ広告だけ載せて公開（アフィリASPは無料サブドメインでも通りやすいものがある）。トラフィックが付いた段階で、**収益が出たツールにだけドメインを買って AdSense を足す**。「当たってから¥1,000払う」ので持ち出しリスクゼロ。

### 4.A.1++ 「静的か動的か」「DB は使えないの？」（あとから決められる・固定されない）
- まず安心してよい点：**サーバーを自分で立てて管理する必要はない**。静的＝「ファイルを置く→CDNが配信」、動的が要るときも「サーバーレス関数」で、いずれもサーバー運用は不要。
- **静的か動的かは、ホスティング時に一度決めて固定する話ではない。** 既定は**静的（無料・高速・bot費用リスクなし）**。ほとんどのミニアプリ（計算/換算/見積）は入力がブラウザ内で完結するので静的で足りる。
- **特定のミニアプリだけ DB/動的が必要になったら、その1本にだけ動的機能を足せる。** Cloudflare の同じ無料基盤上で **Pages Functions / Workers（サーバーレス）＋ D1（SQLite DB）／KV／R2（ストレージ）** が使える。残りのツールは静的のまま。モノレポと `registry.json` は静的・動的の混在を許容する。
- よって本仕様の含み：「**静的優先、ただし DB/動的が要るアプリは将来その都度足せる（既定ではない・必須でもない）**」。DB前提のミニアプリは作らなくてよいが、作りたくなったら作れる。

### 4.A.1+++ 「Heroku は使えないの？」他サービスの整理
- **Heroku は無料枠を2022年11月に廃止済み（復活なし）**。最安でも Eco dyno $5/月（30分で休止＝コールドスタートあり）＋ Postgres $5/月〜。しかも Heroku は**動的アプリ向けの"サーバー"型**で、**無料の静的ツールサイトには不向き**。
- 整理：**静的サイトを無料で量産・公開するなら Cloudflare Pages（または GitHub Pages）が最適**。Heroku/Render 等の"アプリサーバー"型は、レールAの動的アプリを将来作るときに初めて検討すればよい（その時も Cloudflare Workers + D1 で足りることが多い）。

### 4.A.2 サイト構造（1ドメイン × 多数ページ）
- **1モノレポ → 1デプロイ → 多数ページ**。各ツールはルート `…/tools/<slug>` の1ページ。`registry.json` が全ページの真実の源（どのツールが存在し公開済みか）。
- ハブのトップ＝カテゴリ/検索/新着の一覧（v3 §2(c) の「単機能を1つの良質サイトに束ねる」）。これが回遊と内部リンクを生み、薄い誘導ページ群との差になる。
- 必須ページ（About/Contact/Privacy/Terms）は同一ドメイン配下に常設（AdSense審査・E-E-A-T、§9）。

### 4.A.3 公開フロー（新ツールが本番に出るまで）
1. `tool-builder` が `tools/<slug>/` 一式を生成（§5）。
2. `seo-optimizer`／`monetization` がメタ・schema・内部リンク・広告/アフィリ枠・PR表記を付与。
3. `quality-gate`（Opus）が §5.2／§9 を検査 → pass を `registry.json` に記録。
4. push → **CI（GitHub Actions）が静的ビルド** → **Cloudflare Pages の Preview デプロイ**（PRごとに固有URL）。
5. **人間が夜にプレビュー確認 → マージ/承認**（§7.3・§7.5）。
6. マージで **本番デプロイ** → `sitemap.xml` 自動更新 → **Search Console に index 申請**（API資格情報があればエージェントが自動、なければ人間が1クリック）。
7. `registry.json` を `live` に更新（フック `SubagentStop` で自動）。

### 4.A.4 管理・運用
- **ロールバック**: Git の `revert`＋再デプロイ（または Cloudflare ダッシュボードで前デプロイへ戻す）。
- **棚卸し**: `registry.json` で全ツールの状態・URL・品質pass・収益参照を一望。撤退は `status: retired`＋ページを noindex 化（§8）。
- **無料枠の制限と将来の手当て**: Cloudflare Pages 無料は **1プロジェクト 20,000ファイル／月 500ビルド**。数百〜千ツール規模までは十分。さらに増えたら **Pro（$5/月）**、またはビルドを**バッチ化**（夜にまとめて1回デプロイ）して回数を抑える。ファイルは画像を最小化し1ツールあたりの点数を抑える設計にする。

> **要点**: 公開はバラバラの個別サイトではなく、**1ドメイン上の集約サイトに、Gitプッシュ→CIビルド→自動デプロイで各ページを足していく**。固定費は実質ドメイン代（年¥1,000〜2,000）だけ。

### 4.A.5 マネタイズは段階投入（最初は枠だけ → あとで広告）
**広告は最初から貼れない**（AdSense は実在のコンテンツ＋トラフィック＋必須ページが揃ってから審査）。そこで段階を分ける：
1. **Day 1**: レイアウトに**広告枠とアフィリ枠を予約**しておく（あとから実広告を入れても CLS＝表示崩れが出ない設計）。実広告コードはまだ無し（プレースホルダ可）。
2. **早期**: **アフィリリンクを先に設置**（ASP は審査が軽め）。最初の収益はここから。
3. **トラフィックが付いたら**: あなたが AdSense に申請（アカウント・本人/税/支払い・規約同意・サイト審査）→ 発行者ID `ca-pub-…` をエージェントに渡す → **エージェントが予約枠に実広告コードを入れてデプロイ**。
→ あなたが触るのは「申請して ID を渡す」だけ。枠の用意・実コード設置・配置最適化はエージェント（§7.5-B）。`monetization` スキル（§3.3）に「枠予約 → アフィリ先行 → 後から AdSense 注入」の手順を持たせる。

## 5. ユニット（ミニアプリ）の定義

### 5.1 1ユニットの必須要素（=doorway page との決定的な差）
1つのツールは、以下を **すべて** 備えて初めて「公開可能」とする。

1. **動く決定論的ロジック**（`logic.ts`、純関数）と **単体テスト**（`logic.test.ts`）。— これが「機能であって薄い記事ではない」証明。
2. **計算式の出典**（`meta.json` に公式URL）。手数料率・税率・規格・換算式など、**公式が公開している式**を実装（当事者性不要、v3 §5-4）。
3. **入力UI と結果表示**（`packages/ui` の部品を使い、CLS を出さない）。
4. **固有の解説**（`content.mdx`）：問題提起 → 計算式の説明 → 使い方/手順（Omni のパターン）。**他ツールからの薄い差し替えは不可**。
5. **固有のFAQ**（最低3問、その業務に即した実質的な内容）。
6. **構造化データ**（`schema.json`：`WebApplication` ＋ `FAQPage` ＋必要なら `HowTo`）。
7. **内部リンク**（同カテゴリの関連ツールへ。集約サイトの回遊性 = v3 §2(c) の"束ねる"）。
8. **マネタイズ要素**：広告枠（不快でない配置）＋関連アフィリリンク＋**PR表記**（ステマ規制必須）。レールに応じて課金導線（レールAのみ）。

### 5.2 品質下限（quality-gate のブロッカー条件 / これを割ったら公開しない）
- **機能性**: `logic.test.ts` が緑（境界値・既知ケースを含む）。計算が実際に正しい。
- **独自性**: `content.mdx`/FAQ が既存ユニットと過度に類似していない（類似度しきい値以下）。地名/語句だけ差し替えた量産でない（§1.3）。
- **出典**: 計算式の公式出典が `meta.json` にある。
- **必須ページ**: サイトに About/Contact/Privacy/Terms が存在（AdSense・E-E-A-T §9）。
- **性能**: Core Web Vitals（LCP/CLS/INP）が基準内。モバイル対応。
- **規制**: `jp-compliance` チェックを pass（§9：投資助言/税務代行/医療保証等に踏み込んでいない）。
- **表記**: アフィリ/広告に PR 表記がある。優良誤認表現がない。

> quality-gate が1つでも fail を返したら、publisher へ進ませない（フックで強制、§3.4）。

## 6. パイプライン（量産の工程：ステージ別の完了定義つき）

issue 1枚 = ツール1本（または1改善）。issue は `backlog/` に置き、front-matter に `stage` と `status` を持つ。オーケストレーターがステージに応じて委譲する。

| Stage | 名称 | 担当 | 完了定義(DoD) |
|---|---|---|---|
| 0 | **バックログ管理** | orchestrator | issue が テンプレ（§付録）通りに存在し、レール（A/B）と業界が明記。 |
| 1 | **機会発見** | opportunity-scout | 「業界×具体的な××業務」の実ペイン、既存ツールの有無、想定キーワード、**公式計算式の出典URL**、アフィリ在庫がメモ化。抽象カテゴリは不可（v3 §2「具体性」）。 |
| 2 | **仕様化** | orchestrator→tool-builder（`tool-spec`） | 入力/出力/計算式/出典/キーワード/レール/アフィリ先/FAQ案が `tool-spec` テンプレで確定。 |
| 3 | **実装** | tool-builder（`tool-scaffold`） | `tools/<slug>/` 一式が生成され、`logic.test.ts` 緑。**アプリ内にAI/外部API/スクレイピングが無い**ことを確認。 |
| 4 | **SEO** | seo-optimizer（`seo-checklist`） | title/meta/見出し/FAQ/schema/内部リンク/sitemap/canonical が整備。 |
| 5 | **マネタイズ組込** | tool-builder/seo-optimizer（`monetization`） | 広告枠・アフィリリンク・PR表記が設置。CLS を出さない配置。 |
| 6 | **品質ゲート** | quality-gate（`quality-rubric`+`jp-compliance`） | §5.2 を全項目 pass。`registry.json` に pass フラグ。 |
| 7 | **公開** | publisher（`publish-runbook`）＋**人間承認** | プレビュー確認 → 人間がマージ/承認 → 本番デプロイ → index リクエスト → registry を `live` に更新。 |
| 8 | **計測&改善** | analytics-analyst | 公開後の GSC/GA/収益を取得。勝ち筋は強化 issue、負け筋は撤退/放置を `decisions.log` に記録（v3 §0「出荷=検証」）。 |

> ステージ間は **独立コンテキストのサブエージェント** に隔離して回すことで、メインの文脈を汚さずトークンを節約する（高ボリューム出力＝テスト/ビルド/取得 はサブエージェント内で消費）。

## 7. 自動化と運用ループ

### 7.1 1日のループ（memo §3.3 / P4 を実装）
- **朝（〜数分）**: 人間が音声/短文で「やりたいこと」を吐き出す → orchestrator が `backlog/` の issue に整形（音声→issue化）。
- **日中**: エージェントが回す。Claude Code の **background subagents** で実装・SEO・品質ゲートを並走、または `claude -p`（ヘッドレス）をスケジュール起動（GitHub Actions / cron）。**破壊的・公開・課金は自動実行しない**（§7.3）。
- **夜（〜30分）**: 人間がレビュー。orchestrator の要約を見て、プレビューを確認し、**マージ/公開を承認**。承認されたものだけ publisher が本番化。

### 7.2 スケジュール実行の選択肢
- **手動セッション中心**（初期推奨）: 夜のレビュー時にまとめて回す。型が固まるまではこれで十分。
- **半自動**: GitHub Actions で `claude -p "backlog の stage<=6 を進めてプレビューまで作り、要承認項目をまとめよ"` を定期実行 → PR を作らせ、人間は PR をレビュー&マージ。
- **エージェントチーム**（任意・上級）: 並列度を上げたい場合に `agent teams` で複数ワーカーを独立コンテキストで走らせる。トークン消費は増えるので予算と相談。

### 7.3 セーフティ境界（自動実行してはいけない＝必ず人間承認）
以下は **エージェントに自動実行させず、人間が承認/実行**する（フックでブロック、publisher の権限を絞る）：
- 本番デプロイ / 独自ドメインでの一般公開
- 課金（Stripe）の有効化・価格変更（レールA）
- 権限・共有設定・アクセス制御の変更、秘密情報の取り扱い
- 外部への投稿・送信（Search Console の index 申請を除き、SNS等への自動投稿はしない）
- アフィリ/広告アカウントの規約同意・設定変更

### 7.4 トークン予算管理（P8）
- 探索・調査系（opportunity-scout, Explore）は **Haiku**。実装・SEO・品質は **Sonnet**。
- 高ボリューム出力（テスト実行ログ・ビルド出力・データ取得）は **サブエージェントに隔離**し、要約だけメインへ返す。
- スキルは Progressive disclosure（重い参照は `references/`、実行は `scripts/`）で常時ロードを避ける。

### 7.5 人間が一度だけやること（最小セットアップ）と自動化の境界

> 方針：**あなたの手作業は極小化し、コードでできることは全部エージェントにやらせる。** ただし「アカウント作成・本人確認/税/支払い情報・規約同意・審査承認」は、各社が本人性を要求する設計であり、かつ自律エージェントに任せるべきでない領域（資格情報・課金・規約同意）なので、**ここだけは本人**が担う。下表がその"避けられない最小リスト"。

**A. あなたが一度だけやること（セットアップ）**

| # | やること | なぜ本人か | 頻度/費用 |
|---|---|---|---|
| 1 | ドメイン取得（Cloudflare Registrar / お名前.com 等）＋ DNS を Cloudflare に向ける | 購入・支払いは本人 | 年次更新・年¥1,000〜2,000 |
| 2 | GitHub アカウント＋空リポジトリ作成 | アカウント作成 | 一度きり・無料 |
| 3 | Cloudflare アカウント作成＋ Pages にリポジトリ接続 | アカウント作成 | 一度きり・無料 |
| 4 | **Google AdSense**: 登録 → 本人/住所/税/支払い情報 → ポリシー同意 → サイト追加＆審査 | 本人確認・税・規約同意・審査 | 一度きり（審査は本物の流入＋必須ページが前提、§9） |
| 5 | アフィリ/ASP（A8.net・もしも・バリューコマース・Amazonアソシエイト・楽天 等）登録 | 本人確認・口座・規約同意 | 使う分だけ一度きり |
| 6 | GA4 と Search Console のプロパティ作成・ドメイン所有権確認、AdSense↔GA4 / GSC↔GA4 を連携 | プロパティ作成と"確認"クリック | 一度きり・数クリック |
| 7 | 取得したIDをエージェントに渡す（GA4測定ID `G-…`、AdSense発行者ID `ca-pub-…`、GSCプロパティ）。読み取りAPIの資格情報は本人が発行し **CIのSecretsに保存**（コードに直書きしない） | 資格情報の発行 | 一度きり |

→ まとめると、あなたの実作業は〈**アカウント約5個を一度作る＋ドメイン1個買う＋IDを数個貼る＋審査/確認ボタンを数回押す**〉。以降の運用はエージェントが回し、あなたは夜のマージ承認（§7.3）だけ。

**B. エージェント（Claude Code）がやること（あなたは触らない）**
- AdSense の広告スクリプト・広告ユニットをコンポーネントに設置（CLS を出さない配置）。
- `ads.txt`・`sitemap.xml`・`robots.txt`・所有権確認用ファイル/タグをリポジトリに生成。
- GA4 タグ埋め込み、構造化データ(schema)、内部リンク、メタ整備。
- ビルド／デプロイ（CI）／index 申請（資格情報があれば自動）。
- 計測の取得・集計・レポート・改善提案（§8）。

> **重要な線引き**: エージェントは「あなたの代わりにアカウントを作る／税・支払い・本人情報を入力する／規約に同意する」ことは **しない（させない）**。そこはフックとセーフティ境界（§7.3）で止め、必ず本人に渡す。これは安全のためであり、同時に各社の審査・規約上もそうあるべき。

## 8. データモデルと計測（KPI）

### 8.1 `registry.json` スキーマ（全ユニット台帳）
```jsonc
{
  "tools": [
    {
      "slug": "yutai-cross-cost",
      "title": "優待クロス コスト計算＋権利日カレンダー",
      "industry": "個別株投資家",
      "task": "優待クロスのコスト最適化・現渡し管理",
      "rail": "B",                       // "A" | "B" | "B(A)"
      "formula_sources": ["https://..."],// 公式の計算式出典
      "affiliate_targets": ["証券口座"],
      "status": "live",                  // backlog|building|qa|preview|live|retired
      "quality_gate_passed": true,
      "url": "https://hub.example.com/tools/yutai-cross-cost",
      "published_at": "2026-06-20",
      "metrics_ref": "metrics/2026-06.json#yutai-cross-cost"
    }
  ]
}
```

### 8.2 計測対象（analytics-analyst が取得・集計）
- **GSC**: impressions / clicks / 平均掲載順位 / index 状態（ツール別）。
- **GA4**: users / engagement / 入口ページ。
- **収益**: AdSense（RPM/収益）＋ ASP（クリック/承認/報酬）。

**取得の仕組み（ページ別の閲覧数・収益をどう取るか）**
- 全ページに **GA4 タグ**を入れる（測定ID `G-…` は §7.5-7 で人間から受領）。→ ページ別の閲覧数・流入元・回遊が取れる。
- **AdSense ↔ GA4 を連携**すると、GA4 上で**ページ別の広告収益・RPM**が見える（あなたの懸念どおり「このページからいくら上がったか」が取れる）。
- **GSC ↔ GA4 を連携**で、ページ別の検索表示回数・クリック・掲載順位を GA4/レポートに統合。
- `packages/analytics` が各API（GSC API・GA4 Data API・AdSense管理API／ASPのレポート）から取得し、`metrics/YYYY-MM.json` にスナップショット保存。資格情報は **CIのSecrets**（コードに直書きしない）。
- `analytics-analyst`（§3.2）がこのスナップショットを読み、ツール別の勝ち筋/負け筋を判定して改善・撤退の issue を生成（§8.3）。
- 補助として Cloudflare Pages の無料アクセス解析も併用可（プライバシー重視・即時）。

### 8.3 工場 KPI（ダッシュボード化）
- 公開本数、index 済み率、合計オーガニックトラフィック、サイト全体 RPM。
- ツール別：流入・収益・順位。**勝ち筋**（伸びている）→ 強化（関連ツール追加=横展開の種、B→A 検討）。**負け筋**（一定期間で流入ゼロ）→ 放置 or 撤退を `decisions.log` に記録。
- 「出荷=検証」の閉ループ：検索順位・CV が市場の判定（v3 §0,§1）。勝ち筋に資源集中。

## 9. ガードレール（機械的に強制する）

すべて quality-gate（§3.2）＋ `quality-rubric`/`jp-compliance` スキル（§3.3）で **リリース前に自動チェック**。「最終判断は必ず専門家に確認」（memo §6）を前提に、安全側へ倒す。

1. **Google 品質（§1.3 の防御線）**
   - 各ユニットが「動く機能」であること（テスト緑・固有ロジック）。
   - 薄いテンプレ量産/誘導ページでないこと（独自解説・固有FAQ・類似度しきい値）。
2. **AdSense ポリシー**
   - 必須ページ（About/Contact/Privacy/Terms）の存在。低価値コンテンツ回避。**人工トラフィック禁止**（本物のオーガニックのみ）。自動生成の薄い記事を量産しない。
3. **アフィリエイト（日本）**
   - **ステマ規制（2023〜）**: PR/広告であることの明示（disclosure）を必須化。
   - 景表法：優良誤認・有利誤認表現の回避。
   - 主な現金化先は ASP（A8.net / もしも / バリューコマース 等）・Amazon アソシエイト・楽天・Google AdSense を `references/jp-asp-list.md` に整理（在庫=高単価案件の有無は出荷時チェック、v3 §4）。
4. **日本の業法（独占業務・助言に踏み込まない、memo §6）**
   - 投資/株式系: **発注しない・特定銘柄の売買助言をしない・誰でも買えるサブスクで情報提供に留める**。→ ツールは **計算・管理・記録・通知** に限定。自動売買/有償シグナル配信は不可。
   - 会計/税務・行政: 税理士/行政書士の独占業務（税務代理・申告書類作成・税務相談・許認可申請代行）に踏み込まない。→「集計・記録・データ出力」に留め、「申告書を作る/申請を代行する」とは謳わない。
   - 法令対応を謳うツール（電帳法等）の要件充足保証はしない。
   - 医療/法律など人の不利益に直結する領域の「保証」を避ける（v3 §6）。

## 10. 構築の進め方（Claude Code への作業フェーズ指示）

**一気に全自動工場をフルスクラッチで作らない。** 以下の順で、まず1本通して"型"を作り、それを部品化・自動化する。

- **Phase 0 — モノレポ＋1本目を手で貫通（型作り）**
  最初の業界は **当事者領域＝投資**（v3 §7）。理由は「投資が特別だから」ではなく、当事者ゆえに **型（B群ツール → SEO/アフィリ導線 → UI部品 → 量産パイプライン）を最速で作れる**から。
  1. `factory/` モノレポ雛形（apps/hub, packages/ui・seo・monetization・analytics, .claude/, CLAUDE.md, registry.json）。
  2. 1本目ツール（例：**優待クロス計算＋権利日カレンダー**、レールB）を §5 の構成で手作りし、§6 の全ステージを人手で通す。必須ページも作る。
  3. デプロイ（Vercel）と index 申請、計測接続まで通す。
  **→ Phase 0 の成果物 = 再利用可能な"工場テンプレート"。これが本命の最初の納品物。**

- **Phase 1 — サブエージェント＆スキルの定義**
  Phase 0 で手作業した工程を `.claude/agents/` と `.claude/skills/` に部品化（§3.2/§3.3）。フックで品質ゲートとセーフティ境界を強制（§3.4）。

- **Phase 2 — パイプライン自動化（型を固める）**
  2〜3本目を半自動で出し、ステージ間委譲・registry更新・夜レビュー要約を安定化。背景サブエージェント or `claude -p` のスケジュール実行を整備（§7）。

- **Phase 3 — 横展開（量産の本番）**
  当事者でない業界（物販/飲食/ハンドメイド/建設/ペット…、v3 §4 の表を"種"として更に棚卸し）へ複製。`opportunity-scout` で機会を量産 issue 化し、工場で回す。勝ち筋には同業界の **レールA（有料）を被せる**（B→A、§2 P5）。
